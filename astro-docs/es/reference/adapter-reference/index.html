Astro Adapter API

Astro está diseñado para facilitar el despliegue en cualquier proveedor de nube para renderizado on-demand, también conocido como server-side rendering (SSR). Esta capacidad la proporcionan los adapters, que son integraciones. Consulta la guía de renderizado on-demand para aprender cómo usar un adapter existente.

¿Qué es un adapter?

Sección titulada “¿Qué es un adapter?”

Un adapter es un tipo especial de integración que proporciona un entrypoint para el renderizado del servidor en tiempo de petición. Un adapter tiene acceso a la Integration API completa y hace dos cosas:

  • Implementa APIs específicas del host para manejar peticiones.
  • Configura el build según las convenciones del host.

La función setAdapter() acepta un objeto que contiene las siguientes propiedades:

name

Sección titulada “name”

Type: string

Define un nombre único para tu adapter. Se usará para logging.

entrypointResolution

Sección titulada “entrypointResolution”

Type: "explicit" | "auto"
Default: "explicit"

Añadido en: astro@6.0.0

Deprecado

entrypointResolution: "explicit" está deprecado. entrypointResolution: "auto" se convertirá en el comportamiento por defecto, y único, en una futura versión major.

Consulta cómo migrar.

Especifica el método que Astro usará para resolver el server entrypoint: "auto" (recomendado) o "explicit" (por defecto, pero deprecado):

  • "auto" (recomendado): Eres responsable de proporcionar un módulo válido como entrypoint usando serverEntrypoint o, si necesitas mayor personalización a nivel de Vite, usando vite.build.rollupOptions.input.
  • "explicit" (deprecado): Debes proporcionar los exports requeridos por el host en el server entrypoint usando una función createExports() antes de pasarlos a setAdapter() como una lista de exports. Esto soporta adapters construidos con la versión 5 de Astro del Adapter API. Por defecto, todos los adapters recibirán este valor para permitir compatibilidad con versiones anteriores. Sin embargo, no se deben crear nuevos adapters con este valor. Los adapters existentes deberían sobrescribir este valor por defecto con "auto" tan pronto como puedan migrar a la nueva API v6.

Funcionalidades de Astro

Sección titulada “Funcionalidades de Astro”

Las funcionalidades de Astro son una forma de que un adapter le diga a Astro si es capaz de soportar una funcionalidad, y también el nivel de soporte del adapter.

Al usar estas propiedades, Astro:

  • ejecutará validación específica;
  • emitirá información contextual a los logs;

Estas operaciones se ejecutan basándose en las funcionalidades soportadas o no soportadas, su nivel de soporte, la cantidad de logging deseada, y la propia configuración del usuario.

Funcionalidades del adapter

Sección titulada “Funcionalidades del adapter”

Un conjunto de funcionalidades que cambian el output de los archivos emitidos. Cuando un adapter opta por estas funcionalidades, recibirá información adicional dentro de hooks específicos y debe implementar la lógica adecuada para manejar el output diferente.

middlewareMode

Sección titulada “middlewareMode”

Type: MiddlewareMode
Default: "classic"

Añadido en: astro@6.0.0

Determina en qué etapa del ciclo de vida de la página se ejecuta el middleware, y cómo se emite el código del middleware en el output del build.

El modo classic coincide con el comportamiento por defecto de Astro. En las páginas prerenderizadas, el middleware se ejecuta en tiempo de build, y cuando se solicita la página, el middleware no se ejecuta de nuevo. En las páginas dinámicas, el middleware se ejecuta solo en tiempo de petición. El código del middleware es parte de tu server bundle.

En modo edge, el código del middleware puede desplegarse de forma independiente al server bundle, por ejemplo, como una edge function.

Luego, consume el hook astro:build:ssr, que te dará un middlewareEntryPoint, una URL al archivo físico en el sistema de archivos.

buildOutput

Sección titulada “buildOutput”

Type: "static" | "server"
Default: "server"

Añadido en: astro@5.0.0

Te permite forzar una forma de output específica para el build. Esto puede ser útil para adapters que solo funcionan con un tipo de output específico. Por ejemplo, tu adapter podría esperar un sitio web estático para poder crear archivos específicos del host. Por defecto es server si no se especifica.

staticHeaders

Sección titulada «staticHeaders»

Type: boolean
Default: false

Añadido en: astro@6.0.0

Si el adapter proporciona o no soporte para establecer response headers para páginas estáticas. Cuando esta funcionalidad está habilitada, Astro devolverá un mapa de los Headers emitidos por las páginas estáticas. Este mapa está disponible como routeToHeaders en el hook astro:build:generated y puede usarse para generar output específico de la plataforma que controle los HTTP headers, por ejemplo, para crear un archivo _headers para plataformas que lo soportan.

El valor de los headers puede cambiar según las funcionalidades habilitadas/usadas por la aplicación. Por ejemplo, si CSP está habilitado, el elemento <meta http-equiv="content-security-policy"> no se añade a la página estática. En su lugar, su content está disponible en el mapa routeToHeaders.

preserveBuildClientDir

Sección titulada “preserveBuildClientDir”

Type: boolean
Default: false

Añadido en: astro@6.0.0

Cuando es true, los builds estáticos preservarán la estructura de directorios client/server en lugar de outputar directamente a outDir. Esto asegura que los builds estáticos usen build.client para assets, manteniendo consistencia con los builds de servidor.

Esto es útil para adapters que requieren una estructura de directorios específica sin importar el tipo de output del build, como al desplegar en plataformas con requisitos específicos de organización de archivos.

preserveBuildServerDir

Sección titulada “preserveBuildServerDir”

Type: boolean
Default: false

Añadido en: astro@6.4.0

Cuando es true, los builds estáticos preservarán la estructura de directorios client/server para el output del servidor en lugar de outputar directamente a outDir. Esto asegura que los builds estáticos usen build.server para archivos del servidor, manteniendo consistencia con los builds de servidor.

Esto es útil para adapters que requieren una estructura consistente de dist/client/ y dist/server/ sin importar el tipo de output del build, como cuando un sitio estático puede contener aún server islands o image endpoints que necesitan un server entry.

Referencia de tipos del adapter

Sección titulada “Referencia de tipos del adapter”

Los siguientes tipos pueden importarse desde el módulo astro:

AdapterSupport

Sección titulada “AdapterSupport”

Type: AdapterSupportsKind | AdapterSupportWithMessage

Añadido en: astro@5.0.0

Una unión de formatos válidos para describir el nivel de soporte de una funcionalidad.

AdapterSupportsKind

Sección titulada “AdapterSupportsKind”

Type: "deprecated" | "experimental" | "limited" | "stable" | "unsupported"

Define el nivel de soporte para una funcionalidad por tu adapter:

  • Usa "deprecated" cuando tu adapter depreca el soporte para una funcionalidad antes de removerla completamente en una futura versión.
  • Usa "experimental" cuando tu adapter añade soporte para una funcionalidad, pero se esperan problemas o breaking changes.
  • Usa "limited" cuando tu adapter solo soporta un subconjunto de la funcionalidad completa.
  • Usa "stable" cuando la funcionalidad es totalmente soportada por tu adapter.
  • Usa "unsupported" para advertir a los usuarios que pueden encontrar problemas de build en su proyecto, ya que esta funcionalidad no es soportada por tu adapter.

AdapterSupportWithMessage

Sección titulada “AdapterSupportWithMessage”

Añadido en: astro@5.0.0

Un objeto que te permite definir un nivel de soporte para una funcionalidad y un mensaje para loguear en la consola del usuario. Este objeto contiene las siguientes propiedades:

AdapterSupportWithMessage.support

Sección titulada “AdapterSupportWithMessage.support”

Type: Exclude<AdapterSupportsKind, “stable”>

Define el nivel de soporte para una funcionalidad por tu adapter.

AdapterSupportWithMessage.message

Sección titulada “AdapterSupportWithMessage.message”

Type: string

Define un mensaje personalizado para loguear respecto al soporte de una funcionalidad por tu adapter.

AdapterSupportWithMessage.suppress

Sección titulada “AdapterSupportWithMessage.suppress”

Type: "default" | "all"

Añadido en: astro@5.9.0

Una opción para evitar mostrar algunos o todos los mensajes logueados sobre el soporte de una funcionalidad por el adapter.

Si el mensaje de log por defecto de Astro es redundante, o confuso para el usuario en combinación con tu message personalizado message, puedes usar suppress: "default" para suprimir el mensaje por defecto y solo loguear tu mensaje:

También puedes usar suppress: "all" para suprimir todos los mensajes sobre el soporte de la funcionalidad.
